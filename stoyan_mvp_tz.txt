ТЕХНИЧЕСКОЕ ЗАДАНИЕ
на разработку MVP веб-платформы управления и планирования IT-услуг
Проект: "Стоян"

1. ОБЩИЕ СВЕДЕНИЯ

Наименование системы:
Интерактивная платформа управления и планирования IT-услуг для клиентов.

Цель проекта:
Разработать веб-платформу, которая заменяет статичные Excel-таблицы интерактивным интерфейсом для сбора данных клиента, их анализа, формирования плана работ и дальнейшего совместного ведения задач между IT-директором и клиентом.

Формат продукта:
Веб-сервис с разграничением ролей и доступов.

Основная идея MVP:
1. IT-директор создает клиента и отправляет приглашение.
2. Клиент регистрируется и проходит пошаговый сбор данных.
3. Система формирует черновой план на основе введенной информации.
4. IT-директор редактирует и утверждает план.
5. Клиент и IT-директор совместно работают с задачами в едином пространстве.

2. РОЛИ ПОЛЬЗОВАТЕЛЕЙ

2.1. IT-директор
IT-директор является управляющей ролью в системе и работает одновременно с несколькими клиентами.

Права IT-директора:
- просмотр списка всех клиентов;
- создание новой клиентской доски;
- генерация уникальной ссылки-приглашения;
- просмотр введенных клиентом данных;
- просмотр автоматически сформированного плана;
- редактирование задач, сроков, календарей и рекомендаций;
- утверждение итогового плана;
- внесение изменений в план после запуска;
- просмотр актуальных статусов выполнения задач клиентом.

2.2. Клиент
Клиент является представителем бизнеса, который предоставляет исходные данные и далее работает с утвержденным планом.

Права клиента:
- регистрация по уникальной ссылке;
- заполнение исходных данных через пошаговый мастер;
- ручной ввод данных;
- загрузка Excel-файлов для импорта данных;
- просмотр согласованного плана работ;
- изменение статусов задач;
- просмотр календаря и текущего прогресса.

Ограничения клиента:
- клиент не может удалять ключевые блоки плана;
- клиент не может произвольно менять структуру плана без участия IT-директора;
- клиент работает только в пределах разрешенных статусов и задач.

3. ГРАНИЦЫ MVP

3.1. В MVP входит
- личный кабинет IT-директора;
- личный кабинет клиента;
- создание клиента и отправка приглашения;
- регистрация клиента;
- пошаговый мастер заполнения данных;
- ручной ввод данных;
- загрузка Excel-файлов и парсинг данных;
- хранение данных клиента в системе;
- отображение данных в интерфейсе IT-директора;
- формирование предварительного плана системой;
- редактирование плана IT-директором;
- утверждение плана;
- синхронизация плана с клиентским интерфейсом;
- доска задач клиента;
- изменение статусов задач;
- визуализация ключевых показателей в виде дашбордов.

3.2. В MVP не входит
- двусторонняя интеграция с CRM клиента;
- автоматическая постановка задач сотрудникам в сторонние системы;
- сложные AI-модели и LLM-автогенерация как обязательное ядро системы;
- мобильное приложение;
- расширенные сценарии уведомлений и омниканальные коммуникации, если они отдельно не утверждены.

4. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

4.1. Управление клиентами
Система должна предоставлять IT-директору возможность:
- просматривать список всех созданных клиентов;
- открывать карточку или вкладку конкретного клиента;
- создавать новую доску клиента;
- получать уникальную ссылку для регистрации клиента;
- отслеживать этап прохождения онбординга клиентом.

Результат:
Для каждого клиента создается отдельное рабочее пространство с собственными данными, планом, задачами и календарем.

4.2. Регистрация и доступ клиента
После создания новой клиентской доски система должна:
- сгенерировать уникальную ссылку-приглашение;
- позволить клиенту перейти по ссылке;
- открыть форму регистрации;
- сохранить учетные данные клиента;
- зафиксировать согласие с условиями сотрудничества.

Требования:
- ссылка должна быть уникальной;
- доступ клиента должен быть ограничен только его собственными данными;
- один клиент не должен видеть данные другого клиента.

4.3. Первичный сбор данных клиента
После регистрации клиент должен попадать в пошаговый мастер заполнения данных.

Требования к мастеру:
- процесс должен быть разбит на логические этапы;
- количество шагов ориентировочно от 8 до 10;
- пользователь должен понимать, на каком шаге находится;
- обязательные поля должны валидироваться;
- прогресс должен сохраняться, чтобы клиент мог продолжить заполнение позже.

Примерная структура этапов:
1. Финансы.
2. Персонал.
3. Текущая IT-инфраструктура.
4. Операционные процессы.
5. Проблемные зоны.
6. Приоритеты бизнеса.
7. Текущие инструменты и сервисы.
8. Организационные данные.
9. Дополнительные комментарии.
10. Подтверждение полноты данных.

4.4. Ввод финансовых данных
На шаге "Финансы" система должна поддерживать два режима:
- ручной ввод через веб-форму;
- загрузка Excel-файла с последующим разбором данных.

Требования:
- система должна распознавать загруженный Excel-файл;
- данные из файла должны сопоставляться с внутренними полями системы;
- после импорта клиент должен иметь возможность проверить и при необходимости скорректировать данные вручную;
- система должна отрабатывать ошибки формата файла и сообщать пользователю о некорректных данных.

4.5. Ввод данных по персоналу
На шаге "Персонал" клиент должен иметь возможность указать:
- количество сотрудников;
- должности;
- роли;
- распределение ответственности;
- при необходимости дополнительные комментарии по команде.

4.6. Отображение введенных данных
После завершения обязательных шагов:
- клиент должен видеть свою часть платформы с отражением введенных данных;
- IT-директор должен получить доступ к этим данным в интерфейсе конкретного клиента;
- данные должны быть доступны для дальнейшего анализа и формирования плана.

4.7. Дашборд IT-директора
Для каждого клиента в кабинете IT-директора должен быть доступен дашборд.

Дашборд должен содержать:
- ключевые показатели по введенным данным;
- таблицы с детализацией;
- графики и диаграммы;
- календарь мероприятий и дедлайнов;
- блок рекомендаций;
- список задач и этапов.

Система должна поддерживать два режима просмотра:
- режим "Факт";
- режим "План".

Режим "Факт"
Отображает реальные данные, введенные клиентом.

Режим "План"
Отображает автоматически сформированные системой:
- рекомендации;
- предварительные задачи;
- плановые сроки;
- предлагаемые календарные активности;
- целевые значения и ориентиры.

4.8. Формирование плана системой
На основе введенных клиентом данных система должна формировать предварительный план работ.

Для MVP допускается rule-based подход:
- использование заранее заданных шаблонов;
- применение правил в зависимости от отрасли, масштаба бизнеса, структуры расходов, состава персонала и других параметров;
- формирование списка рекомендуемых задач и сроков.

Пример логики:
- при выявлении чрезмерных расходов на ПО система предлагает задачу по аудиту лицензий;
- при нехватке формализации процессов система предлагает задачу по регламентации;
- при отсутствии прозрачности по IT-ролям система предлагает задачи по распределению ответственности.

4.9. Редактирование и утверждение плана IT-директором
IT-директор должен иметь возможность:
- добавлять задачи;
- удалять задачи;
- изменять формулировки задач;
- менять сроки;
- менять приоритеты;
- корректировать календарь;
- править структуру блоков плана до публикации.

После редактирования IT-директор должен иметь кнопку "Подтвердить план".

После нажатия кнопки:
- черновой план становится актуальным рабочим планом;
- клиент получает доступ к утвержденной версии;
- обе стороны работают с одной синхронизированной структурой задач.

4.10. Клиентская доска задач
После утверждения плана клиенту должна открываться интерактивная доска задач.

Допустимые форматы отображения:
- канбан-доска;
- список;
- комбинированный режим, если это допустимо дизайном.

Клиент должен иметь возможность:
- просматривать все этапы плана;
- открывать карточки задач;
- менять статус задач;
- видеть сроки и описание;
- видеть общий прогресс выполнения.

Примеры статусов:
- "Запланировано";
- "В работе";
- "Готово";
- "Отложено".

4.11. Совместная работа с планом
После запуска плана:
- клиент обновляет статусы задач;
- IT-директор видит изменения в актуальном виде;
- IT-директор может вносить структурные корректировки;
- изменения со стороны IT-директора должны отображаться у клиента;
- система должна сохранять целостность плана и разграничение прав.

Желательно для MVP:
- отображать дату последнего изменения;
- отображать автора изменения;
- сохранять минимальную историю обновлений по задачам.

4.12. Календарь
Система должна поддерживать календарное представление:
- сроков задач;
- ключевых событий;
- рекомендованных активностей;
- дедлайнов.

Календарь должен быть доступен:
- в интерфейсе IT-директора;
- в интерфейсе клиента, но только в пределах его проекта.

5. БИЗНЕС-ПРОЦЕССЫ

5.1. Подключение нового клиента
1. IT-директор создает новую доску клиента.
2. Система генерирует уникальную ссылку.
3. Клиент переходит по ссылке и регистрируется.
4. Клиент проходит мастер заполнения данных.
5. Данные становятся доступны IT-директору.

5.2. Формирование плана
1. Клиент завершает ввод обязательных данных.
2. Система анализирует данные.
3. Система формирует предварительный план.
4. IT-директор просматривает режимы "Факт" и "План".
5. IT-директор редактирует предложенный план.
6. IT-директор утверждает план.
7. План становится доступен клиенту.

5.3. Исполнение плана
1. Клиент работает с задачами и меняет статусы.
2. IT-директор отслеживает прогресс.
3. При необходимости IT-директор корректирует структуру или сроки.
4. Клиент получает обновленную версию плана.

6. ТРЕБОВАНИЯ К ДАННЫМ

Система должна обеспечивать хранение следующих сущностей:
- пользователь;
- роль пользователя;
- клиентская организация;
- клиентская доска;
- этап онбординга;
- финансовые показатели;
- сведения о персонале;
- прочие бизнес-данные клиента;
- задача;
- статус задачи;
- дедлайн;
- календарное событие;
- рекомендация;
- версия плана;
- импортированный файл.

Минимальные поля задачи:
- наименование;
- описание;
- статус;
- срок;
- приоритет;
- ответственный;
- источник создания;
- дата создания;
- дата изменения.

7. ТРЕБОВАНИЯ К ИНТЕРФЕЙСУ И UX/UI

Интерфейс должен соответствовать следующим принципам:
- простота и понятность для пользователя без технической подготовки;
- замена сложных Excel-таблиц на визуально удобный веб-интерфейс;
- минимальное количество действий для ключевых операций;
- наглядное отображение прогресса;
- поддержка графиков, диаграмм, таблиц и календарей;
- дружелюбный пошаговый режим ввода данных;
- адаптивное отображение на десктопе и планшетах.

Особое внимание должно быть уделено:
- удобству мастера заполнения;
- понятности статусов задач;
- визуальному разделению режима "Факт" и режима "План";
- простоте навигации между клиентами в кабинете IT-директора.

8. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

8.1. Безопасность
Система должна обеспечивать:
- аутентификацию пользователей;
- безопасное хранение паролей в виде хэшей;
- разграничение прав доступа по ролям;
- изоляцию данных клиентов друг от друга;
- защиту приватных ссылок-приглашений.

8.2. Производительность
Система должна обеспечивать:
- быструю загрузку основных экранов;
- корректную работу при росте количества задач и клиентов;
- обновление статусов без полной перезагрузки страницы, если это возможно в рамках выбранной архитектуры.

8.3. Надежность
Система должна обеспечивать:
- сохранение данных при поэтапном заполнении;
- устойчивую обработку ошибок импорта Excel;
- отказоустойчивую работу ключевых сценариев регистрации, заполнения и утверждения плана.

9. РЕКОМЕНДАЦИИ ПО ТЕХНОЛОГИИ РЕАЛИЗАЦИИ

Этот раздел является рекомендательным и может быть скорректирован перед стартом разработки.

Рекомендуемый стек:
- backend: FastAPI или Django;
- frontend: React или Vue;
- база данных: PostgreSQL;
- контейнеризация: Docker;
- импорт Excel: библиотека для чтения xlsx/xls;
- визуализация: библиотека графиков и календаря;
- API-архитектура с учетом будущих интеграций с CRM.

10. ПЛАН РАЗВИТИЯ ПОСЛЕ MVP

В следующих релизах может быть реализовано:

10.1. Интеграция с CRM клиента
- двусторонняя синхронизация задач;
- создание задач в CRM клиента;
- назначение задач конкретным сотрудникам;
- получение статусов из CRM обратно в платформу.

10.2. Расширенная аналитика
- более глубокие рекомендации;
- отраслевые шаблоны;
- интеллектуальный анализ проблемных зон;
- AI/LLM-механизмы для помощи IT-директору.

10.3. Уведомления
- email-уведомления;
- уведомления в мессенджеры;
- напоминания о сроках;
- уведомления об изменении плана.

11. КРИТЕРИИ ПРИЕМКИ MVP

MVP считается реализованным, если:
- IT-директор может создать клиента и получить ссылку-приглашение;
- клиент может зарегистрироваться по ссылке;
- клиент может пройти мастер заполнения данных;
- клиент может вручную ввести данные и загрузить Excel;
- система сохраняет и отображает введенные данные;
- IT-директор видит данные клиента в дашборде;
- система формирует предварительный план;
- IT-директор может редактировать и утверждать план;
- клиент получает доступ к утвержденному плану;
- клиент может менять статусы задач;
- изменения статусов видны IT-директору;
- доступы ролей разграничены корректно.

12. ВОПРОСЫ НА УТОЧНЕНИЕ ПЕРЕД РАЗРАБОТКОЙ

Рекомендуется отдельно зафиксировать до старта:
1. Какая именно структура Excel-файла будет поддерживаться в MVP.
2. Какие поля являются обязательными на каждом шаге мастера.
3. Нужны ли уведомления уже в первой версии.
4. Нужна ли история изменений задач в MVP или это переносится в следующий этап.
5. Должен ли клиент менять только статусы или также приоритет и комментарии.
6. По каким правилам формируется автоматический план: строго по шаблонам или с элементами AI.
7. Нужны ли роли внутри клиента, кроме одного основного пользователя.
